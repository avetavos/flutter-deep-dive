## จาก code สู่แอปที่ ship จริง

การเขียน widget เป็นแค่ครึ่งเดียวของงาน อีกครึ่งคือเครื่องมือรอบ ๆ มัน: toolchain ที่รัน code ของคุณ, test ที่พิสูจน์ว่ามันทำงาน, build mode ที่แปลง Dart เป็น native binary ที่เร็ว และเรื่อง production ที่ตัดสินว่า user จริงจะมีวันที่ดีหรือไม่

โมดูลนี้คือครึ่งหลังนั้น — ส่วนที่แยก demo ออกจาก product

## โมดูลนี้ครอบคลุมอะไรบ้าง

| บทเรียน | สิ่งที่คุณจะได้เรียน |
|--------|-------------------|
| Project & tooling | `pubspec.yaml`, flutter CLI, DevTools และ hot reload vs hot restart |
| Testing | unit, widget, integration และ golden test |
| Build & release | debug/profile/release mode, JIT vs AOT, flavor, output แต่ละ platform |
| Production concerns | error handling, crash reporting, profiling และ CI/CD |

export const flowDiagram = `flowchart LR
  dev["develop\n(hot reload)"] --> test["test\n(unit/widget/golden)"]
  test --> build["build\n(release, AOT)"]
  build --> ship["ship\n(monitor, profile, CI/CD)"]`;

แต่ละ stage มีเครื่องมือของตัวเองและ failure mode ของตัวเอง เส้นเรื่องหลัก: Flutter optimize หนักมากเพื่อ inner loop ที่เร็ว (hot reload) แล้วสลับไปใช้ compilation strategy คนละแบบเลย (AOT) ตอน release — เข้าใจการแยกนี้แล้วจะเข้าใจ tooling เกือบทั้งหมด

<Callout title="การเปลี่ยนวิธีคิด">
ตอน develop Flutter รัน code ด้วย JIT compiler เพื่อให้ hot-reload การเปลี่ยนแปลงได้ในไม่ถึงวินาที ตอน release มันจะ compile ล่วงหน้าเป็น native machine code เพื่อความเร็วและขนาดที่เล็ก มันคือ runtime สองแบบสำหรับ code เดียวกัน — ซึ่งเป็นเหตุผลว่าทำไม "ทำงานได้ใน debug" ไม่เคยเป็นข้ออ้างเดียวกันกับ "ทำงานได้ใน release"
</Callout>
